### Тяжесть материала — итоговый вердикт

- **HEIC:** браузеры не показывают (только Safari частично) → конвертация **обязательна**.
- **MOV / HEVC:** в Chrome/Firefox без аппаратной поддержки HEVC играет нестабильно → конвертация в MP4 H.264 + WebM VP9 **обязательна**.
- **Видео 60fps + 1080p + ~25–45 МБ за ролик** — слишком тяжело для веба. Цель после сжатия: 5 нужных роликов суммарно **~15 МБ на формат**, итого **~30 МБ** (вместо 139 МБ).

## Решения по дизайну

### Подход к интеграции
**A. Прямая раскладка** (выбрано). Файлы в `public/media/`, пути в компонентах напрямую. Без манифеста и без обёрток-компонентов кроме одного `<VideoPlayer>`.

### Поведение видео
**По клику с постером** (выбрано). До клика грузится только постер-картинка (~80 КБ). После клика — `<video>` с autoplay. Бережёт трафик и батарею мобильных, не блокирует первый рендер.

### Лица врачей
**Обезличить** (выбрано). Невозможно подтвердить, что мужчина с усами и женщина в зелёных скрабах — это именно Хечоян и Навасардян (могут быть другие сотрудники). Поэтому:
- Серые заглушки на профилях врачей и в блоке «Наши врачи» на главной — **остаются**;
- Фото с лицами идут в галерею «О клинике» с подписями «Команда клиники в работе», без имён;
- Видео `khechoyan.mp4` (резерв) тоже без привязки к имени — на сайт пока не вставляем.

## Пресеты сжатия

### Фото (HEIC + JPG → WebP)

```sh
magick input.{HEIC,jpg} -auto-orient -resize {WIDTH}x -quality {Q} output.webp
```

| Группа | Ширина | Quality | Целевой размер |
|---|---|---|---|
| Hero / широкие (`reception-*`) | 1600px | 80 | ~200 КБ |
| Стандарт (`procedures/*`, `doctor-at-work-*`) | 1200px | 80 | ~120 КБ |
| Стоковые (`stock/*`) | 1200px | 80 | ~100 КБ |
| Постеры видео | 1200px | 75 | ~80 КБ |

**Итог по фото:** ~12 файлов, суммарно ≈1.5 МБ.

### Видео (MOV → MP4 H.264 + WebM VP9)

```sh
# MP4 H.264
ffmpeg -i input.MOV \
  -vf "scale='min({W},iw)':-2,fps=30" \
  -c:v libx264 -preset slow -crf 26 \
  -movflags +faststart -an \
  output.mp4

# WebM VP9
ffmpeg -i input.MOV \
  -vf "scale='min({W},iw)':-2,fps=30" \
  -c:v libvpx-vp9 -crf 32 -b:v 0 -an \
  output.webm

# Постер
ffmpeg -ss 1 -i input.MOV -frames:v 1 -vf "scale={POSTER_W}:-1" -q:v 2 poster.jpg
magick poster.jpg -quality 75 poster.webp
```

| Ролик | W | Прогноз размера (mp4 / webm) |
|---|---|---|
| `microscope` | 1080 | ~7 МБ / ~5 МБ |
| `clinic-tour` | 720 | ~2 МБ / ~1.5 МБ |
| `treatment` | 720 | ~3 МБ / ~2 МБ |
| `implant` (резерв) | 720 | ~5 МБ / ~3.5 МБ |
| `khechoyan` (резерв) | 720 | ~2 МБ / ~1.5 МБ |
| **Итого нужных (без резерва)** | — | **~12 МБ / ~8.5 МБ** |

Параметры обоснованы:
- `-an` (без звука) — все ролики беззвучные, фоновые. Экономит 10–20%.
- `fps=30` — на вебе 60 от 30 не отличить, размер падает почти в 2 раза.
- `+faststart` — moov atom в начало, видео начинает играть до полной загрузки.
- 720p вертикали достаточно для мобильных, на десктопе плеер всё равно ~360px шириной.

## Скрипт `scripts/optimize-media.mjs`

Идемпотентный: проверяет mtime источника и таргета, пересоздаёт только устаревшее.

Логика:
1. Читает массив-манифест: `{src, dst, kind: "photo"|"video"|"poster", width, quality}` для каждого файла.
2. Для каждого элемента:
   - Если `dst` не существует или `mtime(src) > mtime(dst)` → запустить `magick` или `ffmpeg`.
   - Иначе пропустить с логом `skip`.
3. После всех — печатает суммарный размер до/после.

Запускается **вручную** из корня проекта: `node scripts/optimize-media.mjs`. Источник по умолчанию — `C:\Users\Roman\Desktop\My-Dentistry\` (захардкожен; для другой машины меняется константой `SRC_ROOT` в начале файла или env-переменной `MEDIA_SRC`). Не часть `npm run build` — ассеты уже в git, повторное сжатие при каждом билде не нужно.

Зависимости: ImageMagick + ffmpeg должны быть в PATH или в стандартных WinGet-путях. Скрипт сам ищет в `C:\Program Files\ImageMagick-*` и `%LOCALAPPDATA%\Microsoft\WinGet\Packages\Gyan.FFmpeg_*\` если в PATH нет.

## Компонент `<VideoPlayer>` (`components/blocks/VideoPlayer.tsx`)

Client component, ~50 строк, без зависимостей.

Props:
```ts
type Props = {
  src: string;          // путь без расширения, напр. "/media/video/microscope"
  poster: string;       // /media/video/microscope.poster.webp
  title: string;        // alt + aria-label
  aspectRatio?: string; // по умолчанию "9/16"
};
```

Поведение:
- До клика — кнопка с постер-картинкой и play-иконкой по центру (overlay `bg-black/20`, hover `bg-black/30`).
- После клика — рендерится `<video controls autoPlay playsInline>` с двумя `<source>` (WebM первым, MP4 вторым).
- `playsInline` обязателен для iOS — иначе раскрывается на весь экран.
- Постер ленится (`loading="lazy"`).

## Образец разметки для `<img>`

```tsx
<img
  src="/media/clinic/reception-detail.webp"
  alt="Приёмная клиники «Моя Стоматология»"
  width={1200}
  height={1600}
  loading="lazy"
  decoding="async"
  className="h-full w-full object-cover"
/>
```

- `width`/`height` обязательны — резервируют место и убирают CLS.
- `loading="lazy"` для всего ниже первого экрана.
- `loading="eager" fetchPriority="high"` — для hero-изображения главной.
- `alt` осмысленный, на русском.

## Что в скоуп НЕ входит

- Не меняем `next.config.ts` (`images.unoptimized: true` остаётся).
- Не добавляем `next/image` — для статического экспорта это лишний слой.

- Не привязываем фото/видео к именам Хечоян/Навасардян — обезличиваем.
- Не добавляем галерею-карусель (отдельный компонент Embla) — в галерее «О клинике» простая сетка.
- Не делаем `<picture>` с AVIF — WebP покрывает >97% браузеров, AVIF добавит ещё один формат для маргинального выигрыша.
